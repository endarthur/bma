1What BMA is

BMA (Block Model Atelier) is a block model inspector that runs entirely in your browser. Drop a CSV exported from Isatis, Vulcan, Surpac, Leapfrog, Datamine or any other mining package and you get, in seconds: descriptive statistics, grid geometry detection, category breakdowns, grade–tonnage curves, directional (swath) profiles, and comparison against the samples the model was estimated from — or against any number of other datasets.

Three properties define the tool:

  • Nothing leaves your computer. There is no server: the file is read locally, by streaming. Safe to use with confidential data.
  • Huge files just work. BMA never loads the whole file into memory — it processes row by row. Multi-gigabyte models run on an ordinary laptop. Speed is bounded by reading the file once, not by RAM: expect a few million rows in seconds and multi-gigabyte files in (low) minutes on a typical machine — wider files (many columns) and calculated columns add to that. A progress bar runs throughout and you can cancel at any time.
  • One pass only. Types, statistics, geometry and calculated columns are computed in a single read of the file, on a separate thread (Web Worker), so the interface never freezes.

BMA is multi-dataset: the block model is the hub, and you compare it against as many other datasets as you need — the composites behind it, an infill campaign, a re-estimate, a second model. Every dataset is a peer with its own import panel, filters, calculated columns and (if it has coordinates) geometry. The address is https://endarthur.github.io/bma/ — and it also works offline after the first visit (see §17).

2Getting started

2.1 The example dataset

The landing page has a “⤓ Download example dataset” button. It generates, in your own browser, a bma-example.zip containing a synthetic block model, the drillhole composites “behind” it, the raw drillhole tables (collar/survey/assays, for §13), and a step-by-step TUTORIAL.txt. It is the best way to get to know the tool: the data was designed so every chart shows something interesting (there is a +2% Fe bias planted in the samples, on purpose).

The example zip is also a packed project: drag the zip itself back into BMA and confirm, and everything opens already configured (see §15).

2.2 Loading your own data

Drag the file onto the dotted zone, or click it to browse. Accepted formats:

  • .csv, .txt, .dat — delimited text (comma, semicolon, tab or space, auto-detected);
  • .zip — with one or more CSVs inside (you choose which one to analyze);
  • .bma.json / .bma.zip — saved or packed projects (§15).

Recently opened files appear as cards on the landing page — in Chromium-based browsers, clicking reopens the file directly, no browsing again.

You don't have to start from a block model. “Start an empty project” opens the workspace with no model at all — useful when you only have drillholes or scattered point data. From there you add datasets (next paragraph), and tabs that need a model (grade–tonnage, the model Δ% reference) simply target the dataset you choose instead. A model-less project saves itself like any other and comes back from the Projects list on the landing page (it has no model file to drop, so that list is how you reopen it); you can also Pack it to a single .bma.zip (§15). You can give the project a model at any time — Load model file… on the Import tab, or drop a model CSV — without losing the datasets you've already added.

Adding datasets. Once a project is open, use ＋ Add dataset at the foot of the data tree (§14), the ＋ launcher on the tab strip, or Data → Add point / drillhole dataset. Each new dataset gets its own import panel; the first comparison is labelled d2, the next d3, and so on. You can also just drop a file onto an open project: BMA asks whether to add it as a comparison dataset or use it as the (replacement) model — it never replaces your data silently.

2.3 Contextual help

On any tab, press F1 (or click the ? in the toolbar) to open that tab's help, with an explanation of every control.

2.4 The workspace: panels that coexist

On wide screens, BMA's tabs are dockable panels. Everything behaves like an ordinary tab bar — until you drag a tab:

  • drop it in the gap between panels (or at the edges) to split the workspace — Statistics next to Swath, GT under Categories;
  • drop it on a panel's body to tear it off into a floating window you can move and resize freely; drop it on a tab strip to re-dock;
  • right-click a tab: Float, Duplicate, Move to new rail, Close, Close others; right-click a floating window's titlebar: Dock / Close.

The analysis panels are also cloneable. Use ＋ on the tab strip (or right-click a tab → Duplicate) to open a second Statistics, Categories, Swath, StatsCat, GT or Export — an independent view with its own selection, directions, reference and target dataset. Put Fe statistics beside SiO₂ statistics, two swaths along different axes, or the model's grade–tonnage beside a comparison dataset's, side by side. Statistics, Categories and StatsCat clones are independent views of the shared analysis; Swath and GT clones run their own worker pass; Export clones each export their own dataset. A duplicate carries the source panel's configuration; each clone is saved with the project.

A fresh workspace opens a focused set — Import Model, Statistics and Categories — and the rest open on demand, so you start with a clean surface instead of a wall of tabs. Closing a panel loses nothing either — its content stays alive. To open or reopen any panel use ⋮ → Panels (open panels show a checkmark), the data tree (§14), or right-click the background of any tab strip. ⋮ → Reset layout returns to that default single-stack arrangement. The arrangement is saved with the project, and the data tree lives in its own collapsible column on the left (◀/▶ buttons).

On narrow screens (phones) none of this changes: BMA keeps the classic tab navigation.

Opening a panel. A fresh workspace shows only Import Model, Statistics and Categories. The other tabs (Calc, GT, Swath, StatsCat, Export) open on demand — from View → Panels (a ✓ marks open ones), from the data tree, or by right-clicking a tab strip. The walkthrough below says “open the GT tab” and so on; that's how. (A packed or saved project restores whatever layout you left, so the example project opens with everything already in place.)

2.5 Your first session: validating the example model

This is the fastest way to understand BMA — a complete model-vs-samples validation on the built-in example, end to end, in about fifteen minutes. The example was built with three defects planted on purpose: the samples carry a +2% Fe assay bias, an infill campaign clusters a quarter of the holes in the rich east (a sampling bias), and the block model is smoother than the point samples. You'll make all three reveal themselves.

Setup. On the landing page click ⤓ Download example dataset, then drag the resulting bma-example.zip back onto BMA and confirm. Everything opens pre-configured. (If you'd rather build it by hand to learn the controls, follow the steps below loading bma-example-model.csv first; open each tab as you reach it, per the note above.)

Step 1 — Read the model before trusting it. On the Import Model panel, after Analyze ▶, switch the right pane to Summary. Confirm what a clean import looks like: 10 m blocks, a 24×24×10 grid, fill ratio 100%, and a green data-health block (no rows dropped). This is your "the file came in correctly" check (§4).

Step 2 — Look at the grade distribution. Open Statistics. Read Fe's mean and CV. Click the Fe name to plot its CDF below, then press Prob and set the X axis to log: a near-straight line means roughly lognormal — your baseline before any comparison (§7.3).

Step 3 — Bring in the samples. The samples are already loaded in the packed project; otherwise add them with Data ▸ Add point dataset and bma-example-samples.csv, set Weight = SUPPORT, and Analyze (§12.1). Back on Statistics, aux:Fe now sits indented under Fe with a Δ% line — and it reads well above +2%. That's the headline, and it's deliberately too big: clustering is inflating it. Steps 6–7 untangle that.

Step 4 — Confirm it, and learn its kind. Click aux:Fe to overlay its dashed CDF (the model's is steeper — support). Then switch the CDF panel to Q–Q with Fe and aux:Fe selected: the cloud runs parallel to and above the 45° line. A parallel offset is additive bias, and the vertical gap is that bias in % Fe (§12.4).

Step 5 — Check it's everywhere, not local. Open Swath, keep direction X, and Generate. The dashed sample series rides above the solid model line in every band — a spatially consistent offset, not an edge artifact. Tick Y and Z too for the other directions (§11, §12.3).

Step 6 — Rule out the drilling pattern. On the samples dataset's panel, open Declustering and Run declustering. The naive mean (~54.3% Fe) drops to ~52.9% declustered — part of the apparent richness was just the infill clustering in high grade. Click Use as weight and re-Analyze (§12.7).

Step 7 — Draw the conclusion. Re-read the Δ% under Fe now that the samples are declustered: it falls to roughly the planted +2%. That residual is the part declustering could not remove — the assay bias. Separating sampling bias (declustering fixes it) from grade bias (it doesn't) is the whole point of the exercise, and the moment BMA's purpose clicks.

Step 8 (optional) — Carry a decision out. Switch the samples panel to Top-cut, drag the red cap line, and use Copy calcol → paste in Calc → Analyze to apply a cap; or use Export to write the cleaned dataset back out (§12.8, §16). Diagnoses flow back into the data.

Now on your own data, a model that passes looks like the mirror image of the example: Δ% near zero across metrics, CDFs differing only by the tails (support), swath series interleaving band to band, and declustering barely moving the sample mean. Anywhere your data departs from that is exactly where to look.

3Import — checking before analyzing

As soon as the file lands, BMA reads the first ~100 lines and shows the Import Model panel (the model is the first dataset; later comparison datasets get their own import panels, §12). Nothing heavy has run yet — this is the moment to check:

  • Types — each column is tagged NUM (numeric) or CAT (categorical). Click the tag to force the type, or untick Skip to exclude the column from the analysis.
  • XYZ — coordinate columns are detected by name (X/EAST/EASTING…); fix them in the selectors if needed. Without XYZ there is no geometry, no swath and no volume-based GT.
  • Block Dimensions (DXYZ) — optional: per-block dimension columns, for sub-blocked models.
  • Block geometry — the grid classification: grid / point / auto. An imported model defaults to grid (a regular block model); switch to point for scattered data that should never be treated as a grid, or auto to let the detected spacing decide. After analysis a badge shows the outcome — ▦ grid with the block sizes, or ∴ points — and notes when an override disagrees with the detection. The choice is saved.
  • Per-column filter — quick expressions like > 0 that pre-filter values of that column (e.g. ignore zero grades).
  • ZIP — if the file is a zip with several CSVs, an entry selector appears at the top.
  • Row filter — the FILTER box at the top of this panel pre-filters the model's rows before analysis (§5).

When you are happy, click Analyze ▶ at the top left.

4Analysis and the Summary view

The analysis runs as a stream, with a progress bar and a cancel button (cancelling restores the last complete result). The Summary is not a separate tab — it lives inside the Import Model panel, which gains a Preview | Summary toggle once an analysis exists (mirroring how every comparison dataset shows its own summary, §12). When the run finishes the panel switches to the Summary view, which shows:

  • row count, column count, processing time and an estimate of result memory;
  • grid geometry: block size in X/Y/Z, origin, extent, coordinate ordering, sub-block detection, and the fill ratio (fraction of the envelope actually filled) — shown whenever the dataset is classified as a grid. A comparison dataset with coordinates gets the same table in its own Summary view (§12);
  • a data-health block: any rows or values dropped for being unparseable, out of range or filtered — BMA never loses data silently, it counts it.

Results are cached locally: reopening the same file with the same configuration restores everything instantly. Any configuration change (filter, calcol, weight, grid mode…) marks the analysis as stale — the Analyze button highlights, asking for a new run.

5The row filter

The FILTER box lives at the top of the model's Import panel (§3). It takes a JavaScript expression over the row object r; only rows where the expression is true enter all the model's statistics and charts. Examples:

r.Fe > 55
r.LITO === 'HEM' && r.Fe > 60
between(r.Z, 820, 860)
r.RATIO > 4        // filters can use calculated columns!

Click Apply (then Analyze) to apply; Clear removes it. Appendix A lists every helper function available (clamp, ifnull, between…). Column names autocomplete as you type.

Expression basics (the same syntax powers filters, local filters and calcols — you only need a handful of pieces):
  • r.Fe reads the column Fe of the current row r. A column whose name has spaces or symbols: r["Cu (ppm)"].
  • Compare with > < >= <=; equal is === and not-equal is !== (two/three characters — a single = assigns, it does not test).
  • Combine conditions with && (and), || (or), ! (not): r.Fe > 55 && r.LITO === 'HEM'.
  • Text goes in quotes: 'HEM' or "HEM". Numbers don't.
  • If/else in one line uses ? : — condition ? valueIfTrue : valueIfFalse (used in calcols, §6).
That's enough for almost everything; Appendix A adds ready-made helpers (between, cap, clamp…) so you rarely need more.

On the Categories tab, unticking values in the table builds this filter automatically for you (§8). Each comparison dataset has its own filter, set on its import panel or from the tree's Filter… action (§12, §14).

6Calc — calculated columns

The Calc tab creates new variables from the existing ones. You write a small code block that adds columns to each row — assign to a new name on the row object r and that name becomes a column:

r.RATIO = r.Fe / r.SiO2;
r.ORE = r.Fe > 55 ? 'ore' : 'waste';
  • Simulate runs the code on the sample rows and shows the new columns detected, with inferred type (NUM/CAT) and a preview table. Errors show immediately.
  • The variable browser (left) inserts r.column with one click; expanding a variable shows min/max/mean or the categorical values.
  • Full JavaScript is allowed: conditionals, loops, strings (r.LITO.startsWith('BIF')).
  • BMA infers each new column's type from its values. To force it (advanced), add the name to the type list: r.META.num.push('name') marks it numeric, r.META.cat.push('name') categorical.
  • After simulating, click Analyze — calcols are computed during the analysis and appear in every tab (tagged CALC), including as candidates for weight, GT grade, swath variable, and so on.

Later calcols can use earlier ones (r.B = r.A * 2), but code order is evaluation order. Avoid naming columns after functions (abs, min…), which would be shadowed.

When more than one dataset is loaded, a dataset picker at the top of the editor switches which dataset the calcols apply to — the model or any comparison dataset (each keeps its own code block); see §12.6.

7Statistics — metrics, weights and CDF

With two or more datasets analyzed, a Dataset picker appears at the top of the sidebar: it chooses which dataset is the panel's primary (its columns are the table rows), and every other dataset — the model included — becomes a Δ% comparison below. By default the primary is the model, so a single-model project looks exactly as described here; the picker just means any dataset can be the anchor (§12.2).

7.1 The table

One row per numeric variable; columns configurable in the sidebar: count, nulls, zeros, min/max, mean, standard deviation, CV%, skewness, kurtosis, and whatever percentiles you want (quartile/decile presets or your own list, e.g. 5,10,25,50,75,90,95). Copy table copies everything as a table to paste into Excel.

Statistics is cloneable (§2.4): open a second one with ＋ → New Statistics (or tab → Duplicate) to keep two independent variable selections — for example one panel for grades, another for contaminants.

7.2 Weights

The Weight selector weights all numeric statistics by a column (or calcol) — for example, ore proportion or density (mass-weighted means). The weight is a role of the variable: the same one drives the Swath tab, and shows as a W badge in the data tree (§14). Details that matter:

  • changing the weight requires a new Analyze;
  • rows with a missing, zero or negative weight are excluded (the counter appears under the selector);
  • with a weight active, std/skewness/kurtosis use the population form (small-sample corrections do not apply to arbitrary-scale weights);
  • enable the ΣW metric to see the total weight next to the count;
  • percentiles and the CDF are weighted too — weight 2 is exactly equivalent to repeating the row twice.

7.3 CDF, Prob and Q–Q

Click the name of any variable in the table to add its cumulative distribution (CDF) curve to the lower panel; click again to remove it. Linear or log scale, a tooltip with per-curve readings, and SVG/PNG buttons to export the chart. The Q–Q mode is described in §12.4 — it compares two distributions quantile by quantile.

The Prob button switches the Y axis to the normal-probability scale (probit), ruled from 0.2% to 99.8% — the classic log-probability paper of geostatistics. How to read it:

  • with X in log, a lognormal distribution plots as a straight line: curvature is departure from lognormality, slope breaks suggest mixed populations;
  • the tails — exactly where capping and high-grade decisions are made — are expanded instead of being glued to 0% and 100%;
  • it works with as many curves as you like, including comparison datasets (dashed), to compare model × samples tails.

Reading it on your own data: a model CDF that sits just inside the samples' (slightly less spread, same centre) is the normal signature of block support. Curves that cross, or differ in the body rather than only the tails, point to a real distribution difference — a domaining, units, or estimation issue worth chasing, not just support.

8Categories — categorical variables

The Categories tab explores categorical columns (lithology, domain, classification…). Select a column in the sidebar to see:

  • a bar chart with counts, percentages and a Pareto line (with the 80% mark);
  • Shannon entropy (H) as a measure of mixing;
  • a value table with search, count and percentage.

Features worth knowing:

  • Colors — click a value's color swatch to customize it; the chosen color applies in every tab (StatsCat, GT…).
  • Custom order — drag the table rows (the handles are always visible); the order is respected in the other tabs.
  • Checkbox filter — unticking values builds the expression and offers to apply it to the model's row filter (an inline Apply bar appears; e.g. hide waste from all statistics).

With a comparison dataset analyzed, this tab gains the model × samples proportion comparison — see §12.5 (the screenshot is there). Categories is cloneable too (§2.4).

As on Statistics, with 2+ datasets analyzed a Dataset picker chooses the panel's primary categorical dataset (the bars); the others overlay as proportion diamonds. The model is the default primary.

9StatsCat — statistics by group

StatsCat breaks the numeric statistics down by a categorical variable (Group By) — mean Fe by lithology, for example. Choose the group and the variables in the sidebar (it may require a new Analyze to include variables not yet grouped) and tick which groups to display.

  • Per-group table — count, mean, std and percentiles for each variable × group combination;
  • Per-group CDF — curves colored by category (using the Categories tab colors), linear/log scale and a manual range;
  • Cross-tab — a count matrix (or row/column %) between two categoricals.

If a weight is active on the Statistics tab, the grouped statistics are weighted too (same analysis).

With 2+ datasets analyzed, a Dataset picker selects which dataset StatsCat breaks down (the model by default) — so you can group a comparison dataset's grades by its own categories.

10GT — grade–tonnage curves

The GT tab sweeps cutoff grades and produces the classic curves of tonnage, mean grade above cutoff, and metal content. Configuration:

  • Grade variables — one curve per ticked variable, each with its unit (%, g/t, ppm…), which drives the metal factor;
  • Density — a column, a constant value (e.g. 2.8 t/m³) or none. Without a density, the “tonnage” is actually volume (m³) — the interface warns you;
  • Weight — an optional per-row multiplier column. When the dataset is not a grid (no block volume), this doubles as the tonnage/volume column: pick it to weight tonnage by an explicit per-row value;
  • Block volume — feature-detected: per-row DXYZ if present, else the grid geometry — but only when the dataset is classified as a grid (§3). A point-classified model, or one with no DXYZ/override, falls back to count-based tonnage (1 per row); the sidebar says so and how to get volume weighting (set to grid, assign DXYZ, set an Override, or pick a Weight column). The geometry volume can always be overridden;
  • Cutoffs — a range (min/max/step) or your own list;
  • Group by — separate curves per category (lithology in the figure), with a picker for which groups to show and a total line;
  • Tonnage unit — t, kt, Mt or a custom divisor.

Hover to read exact values on the crosshair. Each chart has Copy SVG and Download PNG (a light, high-resolution version with the variable name in the title), and a collapsible table with a Copy button.

10.1 Theoretical curve from the samples

The Theoretical (samples) section of the sidebar overlays the theoretical grade–tonnage curve derived from a sample distribution, with an affine support correction: the variance of the point distribution is shrunk by the factor f (slider), simulating block support. It is available whenever GT targets a gridded dataset — the model by default, or any gridded comparison estimate selected with the GT Dataset picker — with a from selector choosing which loaded dataset supplies the samples (you validate one estimate against the samples it was built from, never against itself). Reference points:

  • f = 1 reproduces the samples' empirical GT exactly (no correction);
  • f < 1 shrinks the variance around the mean — less tonnage above high cutoffs, lower mean grades above cutoff: the expected effect of going from point samples to blocks;
  • the curve uses the comparison dataset's active weight (e.g. declustering, §12.7) — compare representative distributions, not ones biased by the drilling pattern;
  • the overlay appears on charts without group-by, scaled to the target estimate's total tonnage.

If the estimate's curve lands reasonably on the theoretical one for a plausible f (typically 0.4–0.8 depending on the support ratio), the estimation smoothing is consistent with the expected change of support. An estimate much “smoother” than the theoretical curve at any f is a sign of over-smoothing — high grades diluted too much.

11Swath — directional profiles

Swath slices the model into bands along a direction and shows how each variable behaves in space — the standard tool for seeing trends and validating estimates.

  • Directions — tick X (W→E), Y (S→N) and/or Z (bottom→top): all ticked directions are computed in a single pass and become tabs in the output. Each direction has its own bin width (default = the block size on that axis; increase it to smooth);
  • Custom — a rotated direction defined by dip direction (0° = north, clockwise), dip and rake, generating the U/V/W frame (U = the rake direction in the plane, W = the pole). For the classic “azimuth + plunge” case, use dip direction = azimuth, dip = plunge, rake = 90°;
  • Statistic — mean ± std, P25/50/75 or P10/50/90 (shaded bands);
  • Weight — weights the primary dataset's band statistics (labelled Weight (model) when the primary is the model, where it is the same weight as the Statistics tab — changing it in one place changes it in the other);
  • Display — toggles for bands, count bars and the table; linear/log Y scale; Overlay layout (everything in one chart, one Y axis per variable) or Split (one panel per variable);
  • Colors — click the swatch next to the variable.
  • Dataset — with 2+ datasets analyzed, picks the panel's primary dataset (the solid series); every other dataset overlays dashed. The model is the default primary.

A crosshair with per-variable readings, Copy SVG / Download PNG, and a per-band table with copy. Swath is cloneable (§2.4) — and each clone runs its own directions and comparison fan-out, so you can show two independent swaths at once. The screenshot for this tab is in §12.3, already with the samples overlaid — the most valuable use of the swath.

12Comparison datasets — validation

BMA compares the block model against one or more other datasets — typically the drillhole composites that fed the estimate, but also an infill campaign, a re-estimate, or a second model. It is the tool's answer to the central question of validation: does the model honor the data?

Each comparison dataset is a peer with its own import panel. Add one with Data ▸ Add point dataset… or Add drillhole set… (each opens a fresh import tab), or drag a file onto a new dataset tab. The first comparison is historically labelled aux; further ones are d2, d3… Each carries its own coordinates, filter, calculated columns, weight and — when it has XYZ — its own geometry.

A comparison dataset is a separate set of rows, not extra columns on the model: there is no per-block join. The comparison is statistical (means, distributions, proportions) and spatial (swath). But a comparison dataset is otherwise first-class — if it carries coordinates it gets grid geometry, a grid/point override, a bounding box, and OBJ + CSV export from its Summary view (§16), exactly like the model.

12.1 Loading and configuring

  • Loading — drag the file onto the dataset's import panel, or: if the main data came from a zip with several CSVs, use “use an entry from…” to point at another entry; if you drop a zip here, an entry selector appears in the sidebar.
  • Display prefix — just the label of the dataset's variables in the interface (aux:Fe, samples:Fe…). Cosmetic: in code and filters the reference is always the dataset's handle (aux., d2.…).
  • Coordinates — the dataset's X/Y/Z (detected by name: EAST/NORTH/ELEV…). Model and comparison must be in the same coordinate system for the swath to make sense.
  • Block geometry — the same grid/point/auto override as the model (§3); a scattered sample set defaults to auto and is detected as points.
  • Filter — pre-filters the dataset's rows, with the dataset's handle: e.g. aux.SUPPORT > 1.
  • Weight — a weight for all of that dataset's statistics: composite length (in the figure, SUPPORT), any column or calcol, or (declustered weights) — the declustering weights BMA computes itself (§12.7). Missing/≤0 weights exclude the row.
  • Analyze — runs the dataset's full statistical pass. Required for the Statistics/CDF/Q–Q/Categories comparisons (the swath runs its own pass and does not depend on it). Changed a filter/weight/calcol? The status says “re-run Analyze”.

12.2 Statistics and CDF compared

After a comparison dataset's Analyze, its variables appear at the end of the sidebar (pre-ticked when they have a model counterpart). In the table, aux:Fe sits right below Fe — mean against mean, CV against CV — with a Δ% line showing the difference against the reference dataset. In the example, the samples' Fe mean runs ~2% above the model's: the planted bias, visible immediately. Click the aux:Fe name to overlay its dashed CDF: the model's curve is steeper — block support smooths out the tails that point samples still have.

Reading it on your own data: Δ% within a percent or two across all metrics means the model honors the samples in aggregate. A consistent sign on the means of several grades (all model-low or all model-high) suggests a genuine bias; a large CV or skewness gap with similar means is usually support (block averaging) — confirm with Q–Q (§12.4) and a declustered comparison (§12.7) before concluding.

Two controls make the comparison flexible. A Dataset picker (top of the sidebar, shown with 2+ datasets analyzed) sets the panel's primary — the dataset whose columns are the rows; every other dataset, the model included, becomes a comparison. The default primary is the model, so nothing changes for a plain model-vs-samples check; the picker just lets you flip the anchor — make the samples the primary and read the model as the comparison, say. Then, with three or more datasets loaded, a row of dataset chips appears: toggle which comparisons show, and pick the Δ% reference (the denominator) per panel — so one Statistics clone can compare against the samples while another compares against a re-estimate.

12.3 Swath compared

On the Swath tab, each comparison dataset's variables join the end of the list (split off by a separator with the dataset name). The comparison series is dashed, in the same color and on the same Y axis as the model's counterpart variable — deliberately: independent axes would “autoscale” the difference out of the chart, hiding exactly the bias the chart exists to show. The tooltip shows n= per band. Every loaded dataset fans out in the same pass; the dataset chips choose which to draw.

Reading it on your own data: a healthy overlay has the dashed and solid series interleaving band to band. A persistent one-sided gap across most bands is a spatial (conditional) bias; divergence only at the extreme bands — where sample counts are tiny (watch the count bars) — is usually just thin support there, not a model fault.

12.4 Q–Q: the bias, measured

In the CDF panel, the Q–Q button switches to the quantile–quantile plot: the first selected curve becomes the reference axis (X) and the others plot their quantiles against it, from P1 to P99 (bigger dots at the deciles; hover to read values). How to read it:

  • points on the dashed 45° line → equivalent distributions;
  • a parallel offset → bias (the vertical distance IS the bias, in the variable's units);
  • rotation/flattening → a dispersion difference — typically block support shrinking the variance.

It works between any two curves (two model domains, two datasets), not just model × samples. Log–log scale available; exports with the same SVG/PNG buttons.

Reading it on your own data: points on the 45° line = the two distributions match. A parallel offset = additive bias (read its size off the axis). A rotation off the 45° line = a variance (spread) difference: the cloud tilts toward whichever dataset is more spread — point samples are wider than block-support estimates, so a mild tilt is the normal support signature, while a strong rotation is over-smoothing. A kink or S-shape = the agreement changes with grade, often a high-grade capping or top-cut question.

12.5 Categories compared

When a comparison dataset has a same-named categorical column (LITO × LITO), the Categories tab overlays a hollow diamond per value, marking each dataset's proportion on the same scale as the bars. The table gains n / % columns per dataset, and values that exist only in a comparison appear at the end with the comparison-only tag — a logged lithology the model never used is exactly the kind of disagreement worth seeing.

12.6 Calcols on a comparison dataset

The dataset picker in the Calc editor points it at any comparison dataset: columns autocomplete as aux.column (or d2.column…), the preview uses that dataset's rows, and the columns you create apply to everything for it — including as a weight and in the comparisons. If a comparison calcol has the same name as a model one (e.g. RATIO in both), they pair automatically in every comparison (and the pairing can be edited by hand — §14). After editing, run the dataset's Analyze again.

12.7 Declustering

Drilling clusters where grade is high — and the samples' “naive” mean inherits that optimism. The Declustering section of a dataset's sidebar implements GSLIB's cell declustering (DECLUS): space is divided into cells, each sample gets a weight inverse to its cell's occupancy, and a sweep of cell sizes searches for the one that minimizes (or maximizes) the declustered mean.

  • Cell sizes — the sweep range (automatic: from 1/100 to 1/2 of the largest extent) and the number of steps; anis Y/Z stretches the cells for anisotropic drill patterns;
  • Find — minimize (the typical case: infill in high grade) or maximize the declustered mean;
  • Run declustering — runs the sweep and reports: naive mean → declustered mean, optimal cell size, the weight range, and the interactive curve (crosshair to read; click to pin a specific cell size — the pin survives project re-runs);
  • Use as weight — applies the weights as the dataset's weight; the Weight selector switches to (declustered weights). Run the dataset's Analyze again: every comparison now uses the declustered distribution.

In the example dataset, the drill pattern has infill planted in the rich zone on purpose: the samples' naive mean (~54.3% Fe) drops to ~52.9% declustered — and the remaining difference against the model is the planted assay bias. Separating sampling bias (declustering fixes it) from grade bias (declustering does not) is exactly what this tool is for.

The weights are never saved in the project — only the parameters. After reopening, click Run declustering again (the status reminds you). Changing the dataset's filter, calcols or coordinates invalidates the weights automatically.

12.8 Top-cut — capping analysis

The Top-cut button (top of a comparison dataset's main area) swaps the preview for the capping analysis of the chosen variable — the four classic views for deciding a top cut, all over the dataset's exact distribution:

  • the histogram and the log-probability plot — where the tail peels away from lognormal is the natural cut candidate;
  • capped mean × cut — how much the mean gives up as the cut comes down;
  • metal removed × cut — % of total metal cut away.

The red line is the cut: drag it in any chart (or type the value) and the status strip recomputes instantly — capped n, % of samples, mean and CV before/after, metal removed. Raw | Declustered switches to the distribution weighted by the declustering weights (§12.7), and Linear | Log controls the X axis.

Once the cut is decided, Copy calcol copies the ready-made line — e.g. aux.Fe_cap = cap(aux.Fe, 63.91); — to paste into the Calc editor. The capped variable enters the comparisons like any other.

13Drillholes — from raw tables to composites

Real campaigns don't arrive composited: they arrive as collar, survey and interval tables. Every dataset's import panel has a Drillhole set card that takes the three files (or a zip containing them): each lands in its role — detected from the headers — and the column mapping appears filled in, always editable. Because each dataset has its own card, a project can hold several drillhole sets — e.g. exploration composites in one dataset, grade-control in another. Each table also has a small ↓ CSV link on its title, and Download tables (zip) bundles the raw set exactly as imported — handy for moving the cleaned tables into another tool.

13.1 The compositing surface

The card is a small compositing workbench. Above the geometry settings:

  • Dip convention — the amber-highlighted row: BMA detects from the survey whether the file uses positive- or negative-down dips, and the toggle is right there to override. Always check it against how your surveys were exported.
  • Desurvey method — minimum curvature (default), balanced tangential, or tangential (straight segments — useful for matching historic desurveys);
  • Composite length — defaults to the file's most common interval length; Min coverage filters poorly-sampled composites (off by default — nothing is dropped silently, the short tail is counted).

And the three controls that shape how intervals become composites:

  • Break on (splits) — tick any categorical columns; a composite restarts at every contact of those columns, so no composite ever straddles a lithology or zone boundary. Tick more than one and it breaks where any of them changes — no need to hand-build a combined-domain key. Each split column rides into the composite unchanged (it is constant within the composite by construction).
  • Density (mass-weight) — name a density column and grade means become length × density (mass) weighted — the geologically correct average once density exists. The density column itself stays length-weighted, SUPPORT stays the covered length, and intervals with no usable density are excluded from the mass-weighting and counted.
  • Combine rules — per output column, how it aggregates over a composite: numeric mean (default, length/mass-weighted), sum, min or max; categorical majority (by covered length) or first (shallowest). Sensible defaults by type, overridable per column.

Each interval table also carries its own calcols & filter (the Interval calcols & filter box), exactly like a dataset (§12.6): a derived column or a row filter authored here applies to that table on export and on compositing — the natural home a single flat composite never had.

13.2 Merging a second interval table

Assays and lithology usually live in separate downhole tables whose breaks don't line up. Drop a second interval file and it is staged as a merge table: BMA joins it to the first on BHID + FROM/TO by union re-segment — every output segment falls inside at most one interval of each table, so columns are carried with no aggregation guesswork. The composite then carries both tables' columns (grade and domain). A segment with no counterpart on one side is null-filled and counted — never silently dropped — and the merge table has its own pre-merge calcols/filter (derive a simplified domain, or drop QA-flagged rows, before the join).

13.3 Compositing & the result

Composite & load desurveys (arc-correct interpolation), runs the (mass-)weighted compositing, and loads the result as that dataset: XYZ at each composite's midpoint, a SUPPORT column (covered length) already applied as the weight, and every other column riding along. Everything in §12.1–12.8 works the same from there.

The consistency report lists everything that didn't join: orphan BHIDs, FROM ≥ TO, overlapping intervals, holes without surveys (desurveyed vertical, counted), merge gaps… nothing vanishes silently. Below it, a length-distribution histogram shows the support before (the raw interval lengths) against after (the composite SUPPORT), on a shared axis — you see at a glance how compositing regularized the support and where the short, low-coverage tail sits (the population Min coverage targets). The provenance strip keeps Report and Edit & re-composite at hand.

Pack (§15) of a project with drillhole sets carries the raw collar/survey/interval files (and any merge table) + the recipe for each — the composites re-derive on load, never stored (the declustering-weights philosophy). The example dataset ships one set: bma-example-collar/survey/assays.csv, with negative dips on purpose.

14The data tree and context menus

The panel to the left of the tabs is the data tree — the property catalog of everything loaded. A search box at its top filters the tree by variable name as you type. For each dataset (the model and every comparison), variables appear grouped under Coordinates (with the assigned X/Y/Z axes), Grades, Categories and Calculated, each with:

  • a color chip — the series color in the charts (Swath, CDF…). A paired comparison variable inherits its model counterpart's color, on purpose;
  • a unit — %, g/t, ppm… One unit per property, valid in every tab (changing it here changes Swath, GT and Summary at once);
  • roles — function badges: W (the statistics weight), GT density and GT tonnage factor;
  • pairing (comparison variables) — ⇄ Fe shows which model variable it is compared against; a red ⇄ — marks an orphan, which enters no comparison at all.

Click any variable to open the editor: color (palette or hex, with “auto” to return to the default), unit, roles and — on comparisons — the Grouped with (model) selector. Automatic pairing is by equal name (case-insensitive); here you fix the cases where names differ: FE_PCT from the samples paired with the model's Fe, no renaming required. A property is the named quantity shared across datasets; colour and unit live on it, so every member follows. Everything you set is saved in the project.

Right-clicking a dataset header offers Filter… (a per-dataset filter, with an instant filtered-size preview) and opens its import panel. The tree column collapses and restores with the ◀/▶ buttons at its top (on narrow screens, with the button at the left end of the tab bar).

Right-click a variable — anywhere: the tree, the Statistics table or sidebar, the Swath and GT lists, the Categories columns — and the context menu opens with the same actions: Properties… (the editor above, at the cursor), set/clear roles, add/remove a CDF curve, unpair or change the grouping. Wherever a table has a copy button, the menu offers Copy table; on the CDF panel, SVG/PNG export; on Categories values, Set color…. Ctrl+right-click always opens the browser's native menu.

15Projects: save, pack, share

15.1 Automatic saving

The entire configuration — preflight, filters, calcols (for every dataset), each panel's selections and clones, weights, colors, the workspace layout — is saved automatically per file in the browser itself. Close, reopen, drop the same file: everything comes back as it was, including the analysis (which is cached). A model-less project (§2.2) is saved the same way, under its own identity, and reopens from the Projects list on the landing page rather than by dropping a model file; its datasets then await re-supply (re-drop each, or use Pack to carry them).

15.2 Save — the project file

The Save action (File menu, or the toolbar) downloads a name.bma.json with the full configuration (without the data). To use it: drop the .bma.json on the landing page — BMA asks for the matching data file; drop that next and the project applies. Comparison datasets come back as empty import panels awaiting their files (re-drop each, or use Pack to carry them).

15.3 Pack — the portable project

The Pack action bundles data + every comparison dataset + configuration into a single name.bma.zip — the right way to send a whole analysis to a colleague. In the dialog:

  • Title — the project name (shows in the toolbar next to the file and names the zip);
  • Contents — the model (if any) and the configuration always go; the comparison datasets are optional (one checkbox covers them all). A drillhole-derived dataset packs its raw collar/survey/interval files, not the composite — they re-derive on load. A model-less project packs the same way, with no model entry;
  • Compress — enables deflate compression (CSV shrinks 4–6×). On by default for data up to ~500 MB; above that the default is off, because without compression both packing and reopening have zero memory cost at any size. There is no 4 GB limit: the files use Zip64 automatically when needed.

The recipient only needs to drop the zip into BMA: the dialog above asks whether to load the embedded project; on confirming, the model and every comparison dataset come out of the file itself and everything opens configured. (Open as zip ignores the project and opens it as a plain zip.)

16Export — writing CSV, OBJ

The Export tab rewrites a dataset (with its filter applied and calculated columns available) as a new CSV, by streaming — it works for files of any size. A Dataset picker at the top chooses which dataset to export when more than one is loaded (the model by default).

  • tick/untick columns; drag to reorder; click the output name to rename;
  • options: delimiter, line ending, decimal separator, null text and numeric precision;
  • a progress bar with the row count while writing.

Any dataset can be exported. A comparison dataset's Summary view carries an Export section: Export rows (CSV) writes all its columns and calcols (with the dataset's filter applied), and — when it is gridded — the bounding box exports as an OBJ mesh. Points and drillhole-derived datasets emit the same shapes as the model.

Beyond CSV, scattered through the interface there are chart exports (Copy SVG, Download PNG in Statistics/GT/Swath) and table Copy buttons (Statistics, GT, Swath, Categories) that paste straight into spreadsheets.

17Installation, offline use and updates

BMA can be used three ways, from most to least connected:

17.1 In the browser, or installed (recommended)

  • Install as an app — BMA is a PWA: in Chrome/Edge, use the install icon in the address bar. It becomes an app with its own window and works offline after the first visit.
  • Build badge — the code in the bottom-right corner (e.g. c18b3dc) identifies exactly which version is running. Click it to check for updates on the spot.
  • Update banner — when a new version is published, BMA notices on its own (on the next load) and shows the banner above; click Reload to update. None of your data is affected — projects and caches are preserved.

17.2 A local copy: one single file

The whole application is a single HTML file. You can save it (Ctrl+S on the page, or receive it from someone), rename it however you like — bma.html, bma-c18b3dc.html — and open it with a double click, no internet, no installation, no server. Everything works: all the tabs, comparison datasets, weights, packed projects, the built-in example. Automatic configuration saving works in the local copy too.

Local-copy caveats:

  • It never updates itself — it is a snapshot of the version it was saved from. The build badge in the corner says which: compare it with the site's to know whether your copy is behind (clicking the badge shows “n/a”, since there is nowhere to check against).
  • PWA installation and the automatic offline mode are exclusive to the hosted version (a local file already is offline by nature).
  • The interface fonts may fall back to the system monospace when you have no network — cosmetic only.

The local copy is a great plan B for environments with no internet or a locked-down network — and since nothing ever leaves the browser in any of the three forms, privacy is the same in all of them.

17.3 Browser support

The core of BMA works in any modern browser — analysis, every chart and table, comparison datasets, packing and reading .bma.zip projects. A few conveniences rely on the File System Access API, which today only Chromium browsers (Chrome, Edge, Brave, Opera) implement; elsewhere BMA falls back gracefully.

DAppendix: recipes

Short procedures for the common jobs. Each points at the section with the detail.

Validate a block model against its composites

  1. Load the model, Analyze; check Summary (geometry, health) — §3, §4.
  2. Add the composites as a comparison dataset; set the length column as Weight; Analyze — §12.1.
  3. Statistics: read the Δ% under each grade; overlay CDFs — §12.2.
  4. Q–Q the grade vs its sample: parallel offset = bias, slope = support — §12.4.
  5. Swath in X/Y/Z: confirm the offset is spatially consistent — §12.3.
  6. Decluster the composites and re-Analyze: the residual Δ% is the part that is not drilling pattern — §12.7.

Explore a single standalone model (no samples)

  1. Load and Analyze; read Summary — §4.
  2. Statistics for each grade (mean/CV/percentiles, CDF, Prob) — §7.
  3. Categories / StatsCat for domains and grade-by-domain — §8, §9.
  4. GT for the grade–tonnage picture; Swath for spatial trend — §10, §11.

Decide and apply a top cut

  1. On the dataset's panel switch to Top-cut, Load distribution for the variable — §12.8.
  2. Drag the red line to where the tail leaves lognormal on the log-prob plot; watch metal-removed.
  3. Copy calcol, paste in Calc, Analyze — the capped variable flows into every view — §6.

Build composites from raw drillhole tables

  1. Drop collar + survey + assays (or their zip) on a dataset's Drillhole set card — §13.
  2. Check the detected roles, column mapping, and especially the dip convention.
  3. Set composite length and Composite & load; read the consistency report.
  4. Analyze — the composites behave like any comparison dataset from here.

Reconcile two estimates

  1. Load the first model; add the second as a comparison dataset; Analyze both — §12.1.
  2. On Statistics, use the Dataset picker to make either the primary; read Δ% with the other as reference — §7, §12.2.
  3. Q–Q one against the other; Swath both to see where they diverge in space — §12.3, §12.4.

Share a whole analysis

  1. Pack → tick the datasets to include → it writes one .bma.zip — §15.3.
  2. The recipient drops the zip into BMA and confirms; everything opens configured.

EAppendix: interpretation gallery

The example dataset teaches what a problem looks like (a planted bias). To read your own model you also need to know what healthy looks like. Each pair below puts a clean validation (left — model vs unbiased, regularly-spaced samples) next to the example's flawed one (right), in the same view.

Δ% table and CDF

Read it: Δ% near zero on every metric = aggregate agreement. A consistent non-zero sign on the means is a bias to chase (decluster first, §12.7); a gap only in CV/spread with matched means is usually just support.

Q–Q

Read it: on the line = match; parallel offset = additive bias; a tilt off 45° = a spread (support) difference; a kink = grade-dependent disagreement.

Swath

Read it: interleaving series = the model honors the samples in space; a persistent one-sided gap = conditional bias; divergence only in the thin end bands (few samples — watch the count bars) is just support there.
